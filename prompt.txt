## FUNCTIONALITY

### Onboarding (`/start`)
- Normal behaviour: new `user_id` sends `/start` → bot replies asking for norms in "Б Ж У" format (one line, space-separated) → user replies `120 60 250` → bot upserts `users` row, confirms with the numbers echoed back and shows the main menu (📈/📅/⚙️).
- Edge case 1: user in onboarding state sends anything that isn't exactly 3 positive numbers → bot replies with a specific format error and stays in the onboarding state; no DB write.
- Edge case 2: an already-onboarded user sends `/start` again → bot shows current norms and asks for confirmation before overwriting (reuses the "⚙️ Изменить норму" flow); it must never silently overwrite.
- Acceptance: see SKILL.md step 3.

### Macro logging (free-text numeric message — two accepted formats)
- Format A — direct eaten grams: exactly **3** numbers = `Б Ж У` already eaten (e.g. `20 10 40`).
- Format B — product entry: exactly **4** numbers, order `вес(г) Б_на_100г Ж_на_100г У_на_100г` (e.g. `150 20 5 30` = 150 g of a product with 20/5/30 g of Б/Ж/У per 100 g). The bot converts this to eaten grams via `factor = вес / 100; eaten_x = per100_x * factor` for each nutrient **before** anything else happens — from that point on Format B is handled identically to Format A (same rollover check, same compensation calculator, same reply).
- The two formats are disambiguated purely by counting whitespace-separated tokens; no keyword or prefix is required.
- Normal behaviour: user sends a valid Format A or Format B message while not in onboarding/edit state → bot performs the 06:00-Minsk rollover check (see below), computes eaten grams (identity for Format A, weight-scaled for Format B), adds them to today's `daily_log` totals, runs the compensation calculator, and replies with the "📈 Сегодня"-style status.
- Edge case 1: message is not exactly 3 or exactly 4 numeric tokens → reject with a human-readable error that shows both accepted patterns; no DB write.
- Edge case 2 (Format A): any of the 3 tokens is negative → reject; no DB write.
- Edge case 3 (Format B): `вес <= 0` → reject ("вес должен быть больше нуля"); any of the three per-100g values is negative → reject; a per-100g value of `0` is valid (e.g. a fat-free product) and must not be rejected.
- Edge case 4: `kcal_fact >= kcal_norm` **before** this message is processed (i.e., the limit was already exhausted by a prior entry) → bot blocks the addition entirely (regardless of which format was used) and replies "Лимит калорий на сегодня исчерпан!"; no DB write. (An entry that itself crosses the threshold is still recorded — the block applies to the *next* attempt, per ТЗ §3.6.)
- Acceptance: see SKILL.md step 1 (conversion function) and step 4 (integration flow, both formats).

### Day rollover (06:00 Europe/Minsk)
- Normal behaviour: on every incoming request, compute current Minsk time; if it's past 06:00 and the most recent `daily_log` row for the user has `date` < today's Minsk date, archive it (leave as historical row) and create a fresh zeroed row for today before processing the request.
- Edge case 1: request arrives between 00:00–06:00 Minsk → the "current day" is still the previous calendar date; no rollover.
- Edge case 2: bot has been idle for multiple days → rollover only ever creates one new row for "today"; it does not backfill missing days.
- Acceptance: see SKILL.md step 5.

### Product-to-grams conversion (pure function, no I/O)
- Signature: `convert_product_to_macros(weight_g: float, b_per_100: float, j_per_100: float, u_per_100: float) -> MacroTotals`.
- `factor = weight_g / 100`; `eaten_b = b_per_100 * factor`, `eaten_j = j_per_100 * factor`, `eaten_u = u_per_100 * factor`.
- Precondition (enforced by the caller in `handlers.py`, not inside this pure function): `weight_g > 0` and `b_per_100, j_per_100, u_per_100 >= 0`. This function itself assumes valid input and does no I/O or Telegram-specific error handling — validation/rejection happens in the handler before calling it.
- Edge case 1: `weight_g` very small (e.g. `1`) → still computes correctly (`factor = 0.01`), no special-casing.
- Edge case 2: a `per_100` value of `0` → resulting eaten grams for that nutrient is `0`, which is valid and distinct from "missing/invalid".
- The output of this function is fed into the same `daily_log` accumulation and `compensate()` call used for Format A — this function only produces the grams-eaten numbers, it does not touch totals or run compensation itself.
- Acceptance: see SKILL.md step 1.

### Compensation calculator (pure function, no I/O)
- Inputs: `norm_b, norm_j, norm_u` (grams), `fact_b, fact_j, fact_u` (grams, cumulative for the day).
- `kcal_norm = norm_b*4 + norm_j*9 + norm_u*4`; `kcal_fact = fact_b*4 + fact_j*9 + fact_u*4`.
- A nutrient is "exceeded" if `fact_x > norm_x`.
- `excess_kcal = Σ (fact_x - norm_x) * kcal_per_gram_x` over exceeded nutrients only.
- `non_exceeded = nutrients where fact_x <= norm_x`.
- `raw_remaining_kcal = Σ (norm_x - fact_x) * kcal_per_gram_x` over `non_exceeded`.
- `budget_for_non_exceeded = raw_remaining_kcal - excess_kcal`.
- If `budget_for_non_exceeded <= 0`: every non-exceeded nutrient's displayed remaining grams = 0, and the day is treated as calorie-exhausted (next logging attempt is blocked per the rule above).
- Else: `scale = budget_for_non_exceeded / raw_remaining_kcal`; each non-exceeded nutrient's remaining grams = `((norm_x - fact_x) * kcal_per_gram_x * scale) / kcal_per_gram_x`.
- Confirmed business rule: when two of three nutrients are exceeded simultaneously, the single remaining non-exceeded nutrient absorbs 100% of the compensation (this falls out of the formula automatically — it is not a special case).
- Edge case 1: all three nutrients exceeded → no non-exceeded nutrient exists; remaining = 0g for all, day blocked.
- Edge case 2: no nutrient exceeded → `scale` will be ≥ 1 only if `budget_for_non_exceeded >= raw_remaining_kcal`, which is always true when `excess_kcal == 0`; remaining grams equal the raw `norm_x - fact_x` unchanged.
- Acceptance: see SKILL.md step 1 and step 2.

### "📈 Сегодня" view
- Normal behaviour: renders eaten vs. norm per nutrient, flags any exceeded nutrient with "(ПЕРЕБОР!)", shows the compensated remaining grams for non-exceeded nutrients, and shows `kcal_fact/kcal_norm`.
- Edge case: day calorie-exhausted → still render the breakdown, but remaining grams shown as 0 for all, plus the exhaustion message.
- Acceptance: see SKILL.md step 2.

### "📅 Календарь" view (full month calendar)
- Normal behaviour: inline keyboard showing the current month as a calendar grid; days with a `daily_log` entry are visually distinguishable (e.g., a filled marker) from days without one; month navigation via "◀" / "▶" buttons; tapping a day with data shows that day's final Б/Ж/У/kcal totals.
- Edge case 1: tapping a day with no log data → bot replies that no data exists for that date, does not crash.
- Edge case 2: navigating to a month with zero entries → calendar still renders (all days un-marked), navigation still works.
- Acceptance: see SKILL.md step 6.

### "⚙️ Изменить норму"
- Normal behaviour: puts the bot into an edit-state waiting for 3 numbers; on receipt, upserts `users.norm_b/norm_j/norm_u`; does **not** touch `daily_log`.
- Edge case: same format validation as onboarding (edge case 1 there) — invalid input keeps the user in edit-state with an error, no partial write.
- Acceptance: see SKILL.md step 3.

## ARCHITECTURE & CODE QUALITY

### Layer split for this project
```
src/nutribot/
  config.py              # env vars: BOT_TOKEN, SUPABASE_URL, SUPABASE_KEY, TIMEZONE
  domain/
    models.py            # UserProfile, DailyLog, MacroTotals (dataclasses)
    calculator.py        # pure functions: kcal(), compensate(), is_day_exhausted(), convert_product_to_macros()
    rollover.py           # pure function: needs_rollover(now, last_log_date) -> bool
  storage/
    repository.py        # Protocol interface + SupabaseRepository implementation
  bot/
    handlers.py           # Telebot message/callback handlers (thin — delegate to domain); includes the
                           # 3-vs-4-token input parser that routes to Format A (direct) or Format B
                           # (product weight+per-100g, via convert_product_to_macros()) before validation
    keyboards.py           # inline keyboard builders (main menu, calendar grid)
    formatters.py          # render domain results into Telegram message text
    states.py              # onboarding/edit-norm state machine
  api/
    webhook.py             # Vercel entrypoint: Flask app, POST /api/webhook
tests/
  domain/ ...
  bot/ ...
```
No God-objects. No cross-layer calls (handlers never touch Supabase directly — always through `storage.repository`; domain layer has zero I/O and zero Telegram imports).

### Type safety
All public interfaces typed. No `any`/`object` without a justification comment on the same line. `storage.repository` defined as a `Protocol` so domain/tests can use an in-memory fake instead of live Supabase.

### Error handling
Catch at the boundary where the error first occurs (parsing in `handlers.py`, DB errors in `repository.py`). User-facing: human-readable Telegram reply. Internal: structured log with context (user_id, action). No uncaught exceptions that crash the Flask process — the webhook handler must return 200 even on internal errors (Telegram retries on non-200), after logging.

### Idempotency
- `users` upsert on `user_id` (onboarding and norm-edit) — enforced via `ON CONFLICT (user_id) DO UPDATE`.
- Day rollover — enforced via a `(user_id, date)` unique constraint on `daily_log`; rollover creation must be `INSERT ... ON CONFLICT DO NOTHING` so a duplicate webhook retry never creates two rows for the same day.
- Macro-add — Telegram may redeliver the same webhook update; use the Telegram `update_id` (or message `message_id`) as a dedup key stored alongside the log write, or make the add handler check-then-act inside a single DB transaction.

### Context budget
- Offload large tool outputs (test logs, migration SQL) to files; read summaries, not raw dumps.
- Use low-effort mode for read-only/directory-listing steps.
- Delegate the SQL schema migration file generation as one discrete step rather than re-deriving it each iteration.
